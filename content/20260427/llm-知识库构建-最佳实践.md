# LLM + 知识库构建 + 最佳实践

## 1. 引言：为什么这件事如此重要

过去两年，大语言模型（LLM, Large Language Model）已经从“令人惊艳的聊天机器人”演变为企业级智能系统的核心能力。然而，真正落地到业务场景时，团队很快会遇到一个关键问题：**通用模型很强，但它并不了解你的业务知识、内部文档、产品规则和最新数据**。

这正是“LLM + 知识库”成为热门架构的根本原因。

在实际应用中，无论是企业内部问答助手、客服机器人、研发知识检索、合规审查支持，还是销售赋能系统，都离不开一个共同目标：**让模型能够基于可信、可控、可更新的私有知识进行回答**。相比单纯依赖模型参数记忆，知识库方案具备几个明显优势：

- **知识可更新**：不需要重新训练模型，只需更新文档与索引。
- **回答可追溯**：可以返回引用来源，便于审计与验证。
- **降低幻觉风险**：通过检索增强生成（RAG）让模型优先基于事实回答。
- **更适合企业治理**：能做权限控制、版本管理、数据隔离和日志监控。

但另一方面，知识库并不是“把 PDF 扔进向量数据库”这么简单。许多团队在 PoC 阶段效果不错，一到生产环境就暴露问题：检索不准、切片混乱、上下文过长、权限失控、召回质量不稳定、成本高且难维护。

因此，本文将从架构和工程视角系统讲清楚：**如何构建一个可落地、可维护、效果稳定的 LLM 知识库系统，以及在实践中应该遵循哪些最佳实践。**

## 2. 核心概念

在进入实现之前，我们先统一几个关键概念。

### 2.1 什么是 LLM + 知识库

通常所说的“LLM + 知识库”，本质上是一种让大模型访问外部知识的系统设计。最常见的范式是 **RAG（Retrieval-Augmented Generation，检索增强生成）**。

其基本流程如下：

1. 用户提出问题。
2. 系统先到知识库中检索相关内容。
3. 将检索结果作为上下文提供给 LLM。
4. LLM 基于问题和上下文生成最终回答。

因此，这类系统并不是让模型“记住所有知识”，而是让它具备“按需查阅知识并总结回答”的能力。

### 2.2 知识库的核心组成

一个完整的知识库系统通常包含以下几个组件：

#### 1）数据源

知识来源可能包括：

- PDF / Word / PPT
- Markdown / HTML / Wiki
- FAQ / 工单记录
- 数据库记录
- API 返回结果
- 企业内部制度、流程、规范文档

#### 2）文档解析

原始文档往往结构复杂，需要先抽取文本、标题、表格、元数据等信息。解析质量会直接影响后续检索效果。

#### 3）切片（Chunking）

文档不能整篇送去嵌入或检索，通常需要切分成多个语义片段。切片策略决定了“可检索性”和“上下文完整性”的平衡。

#### 4）向量化（Embedding）

把文本转换为向量表示，以便进行语义相似度搜索。向量模型质量越高，语义检索能力通常越强。

#### 5）索引与检索

常见检索方式包括：

- **稠密检索**：基于向量相似度
- **稀疏检索**：如 BM25 关键词检索
- **混合检索**：融合向量和关键词结果
- **重排序（Rerank）**：对召回结果再次排序

#### 6）提示词编排

需要把用户问题、检索内容、系统约束和输出格式组织成 Prompt，交给 LLM 生成回答。

#### 7）评估与监控

知识库不是一次性项目，必须持续评估：

- 是否召回了正确内容
- 回答是否忠于原文
- 是否存在幻觉
- 响应时间和成本是否可接受

### 2.3 RAG 不等于万能

很多团队误以为接入向量数据库后，模型就自动“懂业务了”。事实并非如此。RAG 能解决的是“让模型访问外部知识”，但它不能自动修复：

- 原始文档质量差
- 切片方式不合理
- 检索策略单一
- Prompt 设计不清晰
- 缺少评估闭环

所以，**知识库工程的关键，不只是模型，而是完整的数据与检索链路设计。**

## 3. 分步实现：从 0 到 1 搭建 LLM 知识库

下面给出一个比较通用的建设路径。

### 3.1 第一步：明确业务目标

在写一行代码前，先回答这几个问题：

- 用户会问什么类型的问题？
- 回答是“准确优先”还是“覆盖优先”？
- 是否需要引用来源？
- 是否允许模型进行总结、推断、改写？
- 是否有权限隔离要求？
- 成功指标是什么？

例如：

- 内部知识助手：关注召回准确率和引用可追溯
- 客服机器人：关注答案一致性、响应速度和风险控制
- 研发 Copilot：关注多文档聚合能力和代码/文档上下文理解

### 3.2 第二步：准备高质量知识源

知识库效果通常遵循一个朴素规律：**垃圾进，垃圾出（Garbage In, Garbage Out）**。

建议在入库前做以下清洗：

- 去重：删除重复文档和历史版本冗余内容
- 规范化：统一编码、日期格式、标题层级
- 补充元数据：来源、更新时间、部门、权限、标签
- 拆分附件：把正文、附录、表格、图片说明分开处理

如果文档存在多个版本，建议保留：

- `doc_id`
- `version`
- `updated_at`
- `source`
- `access_scope`

这些元数据对后续权限控制、过滤检索和结果解释非常重要。

### 3.3 第三步：设计合理的切片策略

切片是知识库建设中最容易被低估、却最影响效果的环节之一。

常见策略包括：

#### 按固定长度切片

例如每 500 字或 800 token 切一段，并设置一定重叠（overlap）。

优点：简单、稳定。
缺点：容易破坏语义边界。

#### 按语义结构切片

例如按标题、段落、列表、章节进行切分。

优点：语义完整性更好。
缺点：实现稍复杂，对解析质量要求高。

#### 混合切片

先按文档结构切分，再对长段落做二次切片。这通常是更适合生产环境的方案。

经验建议：

- 段落过短，检索命中后上下文不够
- 段落过长，检索精度下降且浪费上下文窗口
- 一般可从 **300～800 token + 10%～20% overlap** 开始调优

### 3.4 第四步：选择 Embedding 模型与向量存储

Embedding 模型决定了“文本如何映射到语义空间”。选择时重点看：

- 中文效果是否稳定
- 长文本支持能力
- 向量维度和推理成本
- 是否适合领域语料

向量数据库可选择：

- FAISS：本地轻量，适合 PoC
- Milvus：适合大规模向量检索
- Weaviate / Qdrant：工程集成较友好
- PostgreSQL + pgvector：适合已有数据库体系的团队

如果场景中关键词也很重要，建议不要只用向量检索，而是直接规划 **混合检索**。

### 3.5 第五步：构建检索链路

一个更稳健的检索流程通常是：

1. 查询改写（可选）
2. 向量召回 top-k
3. BM25 召回 top-k
4. 结果合并
5. rerank 重排序
6. 选择最终上下文

为什么需要这么复杂？因为真实用户问题经常存在：

- 口语化表达
- 关键词缺失
- 多轮上下文依赖
- 同义词替换
- 需要多个文档联合回答

混合检索 + rerank 通常比单纯向量检索稳定得多。

### 3.6 第六步：设计回答生成策略

Prompt 设计的目标不是“让模型更聪明”，而是“让模型更受约束、更可信”。

推荐在系统提示中明确要求：

- 仅根据提供的上下文回答
- 如果上下文不足，明确说不知道
- 优先引用原文事实，不擅自补充
- 输出中标记来源
- 回答风格符合业务要求

如果业务风险较高，还可增加：

- 敏感问题拒答规则
- 合规免责声明
- 审计日志记录

### 3.7 第七步：建立评估闭环

知识库效果不能只靠主观体验，要建立指标体系。

常见评估维度：

#### 检索层

- Recall@k
- MRR
- NDCG
- 命中率

#### 回答层

- Faithfulness（忠实度）
- Answer Relevance（回答相关性）
- Context Precision（上下文精度）
- Hallucination Rate（幻觉率）

#### 工程层

- 平均响应时间
- 每次调用成本
- 索引更新时间
- 错误率

生产环境中，建议保留一组“黄金问题集”持续回归测试，避免每次改动后效果退化。

## 4. 代码示例

下面用 Python 展示一个简化版的 LLM 知识库流程，包括：

- 文档切片
- 生成向量
- 检索相关片段
- 调用 LLM 生成答案

这里用较通用的伪生产代码风格演示，便于理解整体思路。

### 4.1 安装依赖

```bash
pip install openai faiss-cpu numpy
```

### 4.2 构建基础文档处理与索引

```python
import os
import faiss
import numpy as np
from openai import OpenAI

client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))

# 示例文档
DOCUMENTS = [
    {
        "id": "doc-1",
        "title": "请假制度",
        "content": "员工请假分为年假、病假、事假。年假需要至少提前3个工作日申请。病假需提供相关证明。"
    },
    {
        "id": "doc-2",
        "title": "报销流程",
        "content": "差旅报销需在出差结束后10个工作日内提交。发票抬头必须与公司名称一致。"
    },
    {
        "id": "doc-3",
        "title": "远程办公规范",
        "content": "远程办公需经直属主管审批，并保证工作时间在线。涉及客户数据时必须使用公司VPN。"
    }
]


def chunk_text(text, chunk_size=80, overlap=20):
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end])
        start += chunk_size - overlap
    return chunks


def get_embedding(text, model="text-embedding-3-small"):
    response = client.embeddings.create(
        model=model,
        input=text
    )
    return response.data[0].embedding


chunk_records = []
embeddings = []

for doc in DOCUMENTS:
    chunks = chunk_text(doc["content"])
    for i, chunk in enumerate(chunks):
        record = {
            "chunk_id": f'{doc["id"]}-chunk-{i}',
            "doc_id": doc["id"],
            "title": doc["title"],
            "text": chunk
        }
        chunk_records.append(record)
        embeddings.append(get_embedding(chunk))

embedding_matrix = np.array(embeddings).astype("float32")
vector_dim = embedding_matrix.shape[1]

index = faiss.IndexFlatL2(vector_dim)
index.add(embedding_matrix)

print(f"已索引 {len(chunk_records)} 个文本片段")
```

### 4.3 执行检索

```python
def search_knowledge(query, top_k=3):
    query_embedding = np.array([get_embedding(query)]).astype("float32")
    distances, indices = index.search(query_embedding, top_k)

    results = []
    for idx, distance in zip(indices[0], distances[0]):
        if idx == -1:
            continue
        record = chunk_records[idx]
        results.append({
            "score": float(distance),
            "doc_id": record["doc_id"],
            "title": record["title"],
            "text": record["text"]
        })
    return results


query = "病假怎么申请？"
results = search_knowledge(query)
for r in results:
    print(r)
```

### 4.4 基于检索结果生成回答

```python
def build_context(results):
    context_blocks = []
    for i, item in enumerate(results, 1):
        block = f"[来源{i}] 标题: {item['title']}\n内容: {item['text']}"
        context_blocks.append(block)
    return "\n\n".join(context_blocks)


def answer_with_rag(query, top_k=3, model="gpt-4.1-mini"):
    results = search_knowledge(query, top_k=top_k)
    context = build_context(results)

    system_prompt = (
        "你是企业知识库助手。"
        "请仅根据提供的知识库内容回答问题。"
        "如果知识库内容不足以回答，请明确说明“根据当前知识库，无法确定”。"
        "回答时尽量简洁，并在结尾列出引用来源。"
    )

    user_prompt = f"""
用户问题：{query}

知识库上下文：
{context}

请基于上述内容回答。
"""

    response = client.responses.create(
        model=model,
        input=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt}
        ]
    )

    return response.output_text, results


answer, refs = answer_with_rag("病假怎么申请？")
print("回答：")
print(answer)
```

### 4.5 生产化改进建议

上面的代码只展示了最小可用流程。真正进入生产环境时，建议补充：

- 文档解析器（PDF/HTML/Markdown）
- 基于 token 的切片器
- 元数据过滤
- 混合检索
- rerank 模型
- 缓存层
- 访问权限控制
- 评估脚本与监控埋点

例如，一个更合理的数据结构可以是：

```python
from dataclasses import dataclass
from typing import Dict, Any

@dataclass
class ChunkRecord:
    chunk_id: str
    doc_id: str
    title: str
    text: str
    metadata: Dict[str, Any]
```

这样后续做权限过滤、来源展示、版本控制会容易很多。

## 5. 最佳实践 / 常见陷阱

这一部分是最值得关注的内容。很多系统“能跑”并不代表“好用”，而差距往往体现在这些细节上。

### 5.1 最佳实践一：从检索而不是生成开始优化

如果回答不准，很多团队第一反应是换更大的模型。但在知识库场景中，**问题更常出在检索链路**。

优先排查：

- 相关文档是否被正确切片
- 是否召回到正确内容
- top-k 是否合适
- 是否需要关键词召回
- 是否需要 rerank

经验上，检索质量提升带来的收益通常高于一味升级生成模型。

### 5.2 最佳实践二：保留元数据并善用过滤

文档的元数据不只是“装饰信息”，而是生产级系统的核心能力来源。

常用元数据包括：

- 部门
- 文档类型
- 语言
- 更新时间
- 数据级别
- 权限范围
- 产品线

有了这些信息，就能实现：

- 仅检索当前用户有权访问的内容
- 优先检索最新版本
- 按业务域缩小搜索空间
- 在回答中解释来源与时效性

### 5.3 最佳实践三：混合检索往往优于单一向量检索

向量检索擅长语义相似，但对以下情况不一定稳定：

- 精确术语
- 编号、版本号、报错码
- 法规条款编号
- 特定产品名称

例如用户问“E203 错误码是什么意思”，纯向量检索未必比 BM25 更稳。因此企业场景强烈建议采用：

- 向量检索做语义召回
- BM25 做关键词精召回
- rerank 做最终排序

### 5.4 最佳实践四：Prompt 必须约束“未知即未知”

很多幻觉不是因为模型“坏”，而是因为系统默认鼓励它“尽量回答”。在企业场景中，这非常危险。

Prompt 中应明确写出：

- 不得编造制度、价格、政策
- 依据不足时必须承认不知道
- 引用内容优先于常识推断
- 涉及高风险场景时建议人工确认

### 5.5 最佳实践五：建立可观测性

一个生产级知识库系统应该能回答这些问题：

- 用户问了什么？
- 检索到了哪些片段？
- 哪些片段最终进入上下文？
- 模型输出了什么？
- 哪一步耗时最长？
- 哪类问题失败率最高？

建议记录以下日志：

- query
- rewritten_query
- retrieved_chunks
- reranked_chunks
- final_prompt
- model_response
- latency
- token_usage
- user_feedback

这对调试效果和成本控制至关重要。

### 5.6 常见陷阱一：切片过大或过小

切片过大，问题如下：

- 主题混杂
- 相似度被稀释
- 上下文浪费

切片过小，问题如下：

- 丢失前后文
- 回答不完整
- 需要更多片段拼接

因此，不存在“万能 chunk size”，必须结合文档类型和问题类型进行实验。

### 5.7 常见陷阱二：忽视多跳问题

真实用户经常提出需要跨文档整合的问题，例如：

- “远程办公时请假是否仍需走审批流程？”
- “某产品的退款政策与补贴规则是否冲突？”

这类问题需要多个片段联合支持。如果系统只返回单一片段，很容易答非所问。解决思路包括：

- 增加召回数量
- 使用 rerank 提升互补片段排序
- 对复杂问题做 query decomposition（查询拆解）

### 5.8 常见陷阱三：把知识库当成训练替代品

RAG 很适合“外部知识访问”，但并不总能替代微调（fine-tuning）。

如果你的需求是：

- 固定输出风格
- 标准化分类
- 特定格式抽取
- 稳定工具调用行为

那么可能需要 **RAG + 微调** 或 **RAG + 规则引擎** 的组合，而不是只依赖检索。

### 5.9 常见陷阱四：缺少权限控制

企业知识库最危险的问题之一，不是回答错误，而是**回答了不该看的内容**。

必须在检索前或检索时进行权限过滤，而不是等生成后再“看看能不能遮住”。

安全上应遵循：

- 文档级权限控制
- 片段级继承权限
- 用户身份映射
- 审计日志
- 敏感字段脱敏

### 5.10 常见陷阱五：只做离线评估，不看真实反馈

离线数据集很重要，但永远无法覆盖所有真实问题。生产环境中应引入：

- 点赞/点踩
- “是否解决问题”反馈
- 人工纠错标注
- 高价值失败案例回流

一个优秀的知识库系统，往往不是第一次就做对，而是靠持续迭代不断变好。

## 6. 结论

“LLM + 知识库”已经成为企业落地生成式 AI 的主流路径之一。它的价值不在于让模型变成“全知全能”，而在于建立一个**可访问私有知识、可追溯来源、可持续更新、可治理可评估**的智能系统。

回顾全文，一个真正可用的知识库方案应当关注以下要点：

1. **从业务目标出发**，明确使用场景和成功标准。
2. **重视知识源质量**，先清洗、规范化、补充元数据。
3. **精心设计切片策略**，平衡语义完整性和检索精度。
4. **优先优化检索链路**，采用混合检索与 rerank。
5. **通过 Prompt 和规则约束生成**，降低幻觉与风险。
6. **建立评估与监控闭环**，持续迭代系统效果。
7. **把权限、安全、审计纳入架构设计**，而不是事后补丁。

对于刚开始实践的团队，我的建议是：不要一开始就追求最复杂的架构，而是先构建一个可评估、可观察、可迭代的最小闭环。只要你能明确知道“问题出在哪里”，系统就有办法不断优化。

从长期来看，知识库不会只是一个“聊天入口”，它会逐渐演化为企业知识基础设施的一部分，与搜索、工作流、BI、客服、开发平台和协同系统深度融合。届时，真正有竞争力的不是谁接入了 LLM，而是谁建立了**高质量知识、稳定检索能力和工程化治理体系**。

如果你正在规划企业 AI 助手、智能问答或内部 Copilot，LLM + 知识库不是可选项，而是最值得投入打磨的核心能力之一。
