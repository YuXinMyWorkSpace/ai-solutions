# Prompt Engineering + 知识库构建：从原理到落地的企业级实践指南

## 1. 引言：为什么这件事如此重要

过去两年，大模型（LLM）从“能聊天”迅速演进为“能工作”的生产力基础设施。但在真实业务场景中，很多团队很快会遇到一个关键问题：**模型很聪明，却不一定了解你的业务**。

一个通用模型可能擅长写作、总结、翻译和推理，但它通常并不知道：

- 你的公司内部 SOP 是什么；
- 你的产品参数、定价和交付流程是什么；
- 你的客服标准话术与合规边界是什么；
- 你的技术文档、知识文章和故障案例是什么；
- 你的用户真正关心的问题有哪些。

这正是 **Prompt Engineering（提示工程）** 与 **知识库构建** 结合的价值所在。

简单来说：

- **Prompt Engineering** 负责“如何让模型更好地理解任务并输出高质量答案”；
- **知识库构建** 负责“如何让模型在回答时拥有正确、可追溯、与业务相关的上下文”。

二者结合后，才更接近一个真正可落地的 AI 应用形态：

> 让模型在正确的任务框架下，基于可靠知识进行回答。

这类系统通常出现在以下场景中：

- 企业内部知识问答
- 智能客服与售前助手
- 技术支持与故障排查
- 文档检索与摘要生成
- 合同、规范、制度类内容问答
- 研发 Copilot、运维助手、HR 助手

然而，很多项目失败并不是模型不够强，而是因为：

- Prompt 写得模糊、冗长、不可测试；
- 知识库数据质量差，切分混乱，召回不准；
- 没有建立检索、重排、引用、评估的完整闭环；
- 把“模型能力问题”误认为“数据工程问题”，或者反过来。

本文将从技术和工程落地两个维度，系统介绍 **Prompt Engineering + 知识库构建** 的核心方法、实现步骤、Python 示例以及最佳实践，帮助你从“能跑 Demo”走向“可上线、可维护、可优化”的 AI 应用。

## 2. 核心概念

在开始实现之前，我们先统一几个关键概念。

### 2.1 什么是 Prompt Engineering

Prompt Engineering 指的是：**通过设计输入指令、上下文结构、示例和约束条件，引导大模型生成更准确、更稳定、更符合预期的输出。**

一个高质量 Prompt 往往包含以下要素：

1. **角色定义**：告诉模型它是谁。
2. **任务目标**：明确需要完成什么。
3. **上下文信息**：提供背景、限制和输入数据。
4. **输出格式**：规定回答结构，例如 Markdown、JSON、表格。
5. **约束条件**：例如“不要编造”“如果信息不足就说明”。
6. **示例（Few-shot）**：通过样例演示理想输出方式。

例如，一个普通 Prompt 可能只是：

```text
请回答用户问题。
```

而一个更工程化的 Prompt 会写成：

```text
你是一名企业知识库问答助手。
请严格基于提供的资料回答用户问题。
如果资料中没有答案，请明确回答“根据当前知识库无法确认”，不要自行编造。
回答要求：
1. 先给出简洁结论；
2. 再列出依据；
3. 如有适用条件或限制，请单独说明；
4. 使用中文输出。
```

差别在于，后者更易控制输出质量，也更适合测试和迭代。

### 2.2 什么是知识库构建

知识库构建不是简单地“把文档丢给模型”，而是一个数据工程过程，通常包括：

- 数据采集：PDF、网页、FAQ、数据库、工单、Wiki
- 数据清洗：去噪、去重、结构化、脱敏
- 文档切分：按段落、标题、语义片段拆分
- 向量化：将文本转换为向量表示
- 存储索引：写入向量数据库或检索系统
- 查询召回：根据用户问题找到相关片段
- 上下文组装：将召回结果拼接进 Prompt
- 回答生成：由 LLM 基于上下文作答

这个过程通常被称为 **RAG（Retrieval-Augmented Generation，检索增强生成）**。

### 2.3 Prompt 和知识库的关系

一个常见误区是把 Prompt 和知识库分开理解。实际上，二者是强耦合的：

- 没有好的 Prompt，模型即使拿到了正确资料，也可能答非所问；
- 没有好的知识库，Prompt 再漂亮，模型也只能“凭常识猜”；
- 没有好的检索与上下文组织，系统会出现“召回很多，但回答不准”的问题。

因此，一个成熟系统通常由三层组成：

1. **任务层**：Prompt 定义角色、目标、格式、边界
2. **知识层**：知识库提供可信上下文
3. **推理层**：模型根据任务和上下文生成答案

### 2.4 一个典型架构

可以把整体流程理解为：

```text
用户问题
   ↓
Query 重写 / 规范化
   ↓
向量检索 + 关键词检索
   ↓
结果重排（Re-rank）
   ↓
构建上下文 Prompt
   ↓
LLM 生成答案
   ↓
返回答案 + 引用来源
```

如果要进一步提升质量，还可以加入：

- 查询分类路由
- 多轮对话记忆
- 缓存机制
- 安全审查
- 自动评估与反馈学习

## 3. 分步实现：从 0 到 1 搭建一个可用系统

下面我们以“企业内部文档问答助手”为例，介绍一个典型的落地流程。

### 3.1 第一步：明确场景与目标

在写任何 Prompt 或导入任何文档前，先回答四个问题：

1. **用户是谁？** 内部员工、客服、开发者还是终端客户？
2. **问题类型是什么？** FAQ、流程查询、技术排障、政策解释？
3. **可接受的错误类型是什么？** 宁可拒答，还是允许带概率建议？
4. **结果如何评估？** 准确率、召回率、满意度、工单减少率？

例如：

- 场景：员工查询报销制度
- 目标：基于公司制度文档，输出准确且带出处的回答
- 约束：不允许编造，不确定时必须提示联系财务
- 指标：Top-3 检索命中率、答案正确率、人工复核通过率

这是后续所有设计的基础。

### 3.2 第二步：准备知识源

知识库的数据质量，决定了系统上限。

常见知识源包括：

- 公司 Wiki / Confluence
- 产品手册、API 文档
- FAQ 页面
- 历史工单与客服对话
- PDF 制度文件、合同、规范
- 数据库中的结构化知识

这一步的关键不是“多”，而是“准”和“新”。重点建议：

- 删除重复、过期、冲突文档；
- 保留元数据：标题、时间、部门、版本、链接；
- 对扫描版 PDF 做 OCR 校验；
- 对敏感数据进行脱敏；
- 给文档标记权限范围。

### 3.3 第三步：文档切分（Chunking）

文档不能直接整篇送入模型，否则：

- 上下文太长，成本高；
- 检索粒度太粗，召回不准；
- 不相关内容太多，影响生成质量。

因此要对文档进行切分。常见策略有：

#### 按固定长度切分

优点是简单；缺点是可能截断语义。

#### 按结构切分

根据标题、段落、列表、表格进行切分，适合制度文档和技术文档。

#### 按语义切分

根据句义完整性拆分，效果通常更好，但实现更复杂。

实践中常用策略是：

- chunk 大小：300～800 中文字符
- overlap：50～150 字符
- 尽量保留标题层级
- 每个 chunk 附带元数据

一个好的 chunk 应该具备：

- 语义自洽
- 可独立理解
- 不丢失关键上下文
- 与问题有较高匹配概率

### 3.4 第四步：向量化与索引构建

切分之后，需要把文本编码为向量，以支持语义检索。

典型流程：

1. 选择 embedding 模型
2. 为每个 chunk 生成向量
3. 将向量与元数据一起存入向量数据库

常用向量数据库包括：

- FAISS
- Chroma
- Milvus
- Weaviate
- pgvector
- Elasticsearch / OpenSearch（支持混合检索）

如果是生产环境，建议优先考虑：

- 检索延迟
- 可扩展性
- 过滤能力（按部门、时间、权限）
- 混合检索支持
- 运维成本

### 3.5 第五步：检索策略设计

很多系统效果不佳，根源不在生成，而在检索。

仅使用向量检索常见问题：

- 对术语、编号、版本号不敏感
- 对精确关键词匹配较弱
- 容易召回“语义相似但事实不对”的内容

因此推荐采用 **混合检索**：

- **语义检索**：找“意思相近”的内容
- **关键词检索**：找精确术语、错误码、产品名、编号
- **重排模型**：对候选结果重新排序

例如用户问：

> “员工出差住宿报销上限是多少？”

向量检索可以抓到“出差报销制度”，关键词检索可以抓到“住宿”“上限”所在条款，重排则有助于把最相关片段排到前面。

### 3.6 第六步：构建回答 Prompt

这一步是 Prompt Engineering 的核心。

一个典型的知识库问答 Prompt 可以包含：

- 系统角色
- 回答规则
- 引用资料
- 用户问题
- 输出格式

例如：

```text
你是一名企业知识助手。
请基于“参考资料”回答用户问题，不要使用资料之外的未经验证信息。
如果参考资料不足以回答，请明确说明“当前知识库没有足够信息”。

回答要求：
1. 先给出结论；
2. 再给出依据；
3. 列出引用的文档标题；
4. 若存在限制条件或版本差异，请显式说明。

参考资料：
{context}

用户问题：
{question}
```

这里的关键不只是“让模型回答”，而是：

- 限制信息来源
- 要求不确定时拒答
- 强制结构化输出
- 提升可解释性

### 3.7 第七步：评估与迭代

没有评估，就没有优化。

建议至少建立三类评估：

#### 检索评估

- Top-k 命中率
- MRR
- Recall@k

#### 生成评估

- 是否回答正确
- 是否引用了正确依据
- 是否存在幻觉
- 是否符合格式要求

#### 业务评估

- 用户满意度
- 人工转接率
- 工单解决时长
- 问答覆盖率

此外，建议建设一个 **黄金测试集**：

- 50～200 个高质量问题
- 每个问题对应标准答案与参考文档
- 覆盖常见问法、边界问法、模糊问法、错误问法

每次改 Prompt、切分策略、embedding 模型、检索参数时，都跑一次回归测试。

## 4. 代码示例

下面给出一个简化版 Python 示例，演示如何构建一个基础的知识库问答流程。为了便于理解，这里使用本地示例文本、`sentence-transformers` 生成向量，并用 `faiss` 做相似度检索。

> 提示：生产环境通常会替换为更适合中文和业务场景的 embedding 模型，以及托管式向量数据库与 LLM API。

### 4.1 安装依赖

```bash
pip install sentence-transformers faiss-cpu numpy openai
```

### 4.2 准备文档与切分

```python
from dataclasses import dataclass
from typing import List

@dataclass
class DocumentChunk:
    text: str
    source: str
    chunk_id: int


def chunk_text(text: str, source: str, chunk_size: int = 300, overlap: int = 50) -> List[DocumentChunk]:
    chunks = []
    start = 0
    chunk_id = 0

    while start < len(text):
        end = start + chunk_size
        chunk = text[start:end]
        chunks.append(DocumentChunk(text=chunk, source=source, chunk_id=chunk_id))
        chunk_id += 1
        start += chunk_size - overlap

    return chunks


raw_docs = {
    "报销制度.md": "员工出差期间，住宿费按照城市等级执行报销上限。一线城市单晚住宿标准不超过500元，二线城市不超过350元。超出标准部分需经部门负责人审批。交通费需附发票。",
    "差旅流程.md": "员工出差前需在系统提交出差申请，经直属主管审批后方可出行。出差结束后5个工作日内提交报销单据。",
    "财务FAQ.md": "如遇特殊客户接待或展会期间酒店价格异常上涨，员工可提交情况说明，并附审批记录申请例外处理。"
}

all_chunks = []
for source, text in raw_docs.items():
    all_chunks.extend(chunk_text(text, source))

print(f"总 chunk 数: {len(all_chunks)}")
```

### 4.3 构建向量索引

```python
import faiss
import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-small-zh-v1.5")

texts = [c.text for c in all_chunks]
embeddings = model.encode(texts, normalize_embeddings=True)
embeddings = np.array(embeddings).astype("float32")

index = faiss.IndexFlatIP(embeddings.shape[1])
index.add(embeddings)

print("向量索引构建完成")
```

### 4.4 实现检索

```python
def retrieve(query: str, top_k: int = 3):
    q_emb = model.encode([query], normalize_embeddings=True)
    q_emb = np.array(q_emb).astype("float32")

    scores, indices = index.search(q_emb, top_k)
    results = []
    for score, idx in zip(scores[0], indices[0]):
        results.append({
            "score": float(score),
            "text": all_chunks[idx].text,
            "source": all_chunks[idx].source,
            "chunk_id": all_chunks[idx].chunk_id,
        })
    return results


query = "员工出差住宿报销上限是多少？"
results = retrieve(query)
for r in results:
    print(r)
```

### 4.5 构建 Prompt 并调用 LLM

```python
from openai import OpenAI

client = OpenAI()


def build_context(results):
    parts = []
    for i, r in enumerate(results, 1):
        parts.append(
            f"[资料{i}] 来源: {r['source']}\n内容: {r['text']}"
        )
    return "\n\n".join(parts)


def answer_question(question: str):
    results = retrieve(question, top_k=3)
    context = build_context(results)

    system_prompt = """你是一名企业知识库问答助手。
请严格基于提供的参考资料回答问题，不要编造。
如果资料不足，请明确说明“根据当前知识库无法确认”。
回答格式：
1. 结论
2. 依据
3. 引用来源
4. 备注（如有例外或限制）"""

    user_prompt = f"""参考资料：
{context}

用户问题：
{question}
"""

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ],
        temperature=0.2,
    )

    return response.choices[0].message.content


print(answer_question("员工出差住宿报销上限是多少？"))
```

### 4.6 进阶：加入引用与可追溯性

在生产环境中，建议不要只返回“答案”，还要返回引用信息，以增强可信度。

例如可以让模型输出 JSON：

```python
import json


def answer_question_as_json(question: str):
    results = retrieve(question, top_k=3)
    context = build_context(results)

    system_prompt = """你是一名企业知识库问答助手。
请严格基于参考资料回答。
若信息不足，answer 字段写“根据当前知识库无法确认”。
输出必须是 JSON，格式如下：
{
  "answer": "",
  "reasoning": [""],
  "sources": [""],
  "notes": [""]
}
"""

    user_prompt = f"参考资料：\n{context}\n\n用户问题：\n{question}"

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ],
        temperature=0,
        response_format={"type": "json_object"}
    )

    return json.loads(response.choices[0].message.content)
```

这样做的好处是：

- 前端更容易展示结构化结果；
- 可以记录“回答引用了哪些来源”；
- 更容易做自动评估与审计。

## 5. 最佳实践与常见陷阱

在实际项目中，Prompt Engineering 和知识库构建最容易“看起来简单，做起来复杂”。下面是一些经过大量实践验证的经验。

### 5.1 最佳实践一：把 Prompt 当作代码来管理

很多团队把 Prompt 直接写在代码字符串里，最后变成无法维护的“提示词泥团”。

更好的做法是：

- 将 Prompt 模板版本化管理；
- 为每次修改记录变更原因；
- 维护测试样例与预期输出；
- 区分 system prompt、task prompt、format prompt。

建议像管理 API 一样管理 Prompt。

### 5.2 最佳实践二：优先优化检索，再优化生成

如果召回内容本身不对，再强的模型也很难回答正确。

优化顺序建议：

1. 数据清洗
2. 切分策略
3. 检索召回
4. 重排
5. Prompt 结构
6. 模型参数

这是因为大多数知识问答系统的问题，本质是“没找到对的资料”，而不是“模型不会说”。

### 5.3 最佳实践三：让模型学会拒答

企业应用中，“不知道就说不知道”通常比“编了个看似合理的答案”更有价值。

在 Prompt 中要明确加入：

- 如果资料不足，必须说明无法确认；
- 不允许基于常识补全制度细节；
- 冲突资料存在时，要指出冲突并提示人工确认。

这会显著降低幻觉风险。

### 5.4 最佳实践四：保留元数据与引用来源

只存文本，不存来源，是知识库工程中的常见错误。

至少建议保留：

- 文档标题
- URL / 文件路径
- 更新时间
- 版本号
- 业务分类
- 权限标签

这样做的好处在于：

- 便于溯源
- 支持权限控制
- 支持按版本过滤
- 可展示引用增强信任

### 5.5 最佳实践五：构建评估闭环

不要用“感觉回答还不错”来判断系统质量。

建议建立：

- 固定测试集
- 自动化回归脚本
- 标注平台或人工审核流程
- 在线反馈收集机制

一个成熟的团队通常会持续追踪：

- 哪些问题经常答错；
- 哪些文档经常被召回但没帮助；
- 哪类查询需要重写或路由；
- 哪些回答虽然正确，但不够可用。

### 5.6 常见陷阱一：文档切分过大或过小

- **过大**：一个 chunk 包含太多主题，召回噪声高；
- **过小**：上下文碎片化，信息不完整。

解决方案：

- 优先按结构切分；
- 对 FAQ、制度文档、API 文档采用不同切分策略；
- 用测试集验证 chunk 大小对召回效果的影响。

### 5.7 常见陷阱二：过度依赖单一向量检索

对于包含以下信息的场景，仅靠语义检索往往不够：

- 错误码
- 产品型号
- 合同编号
- API 路径
- 版本号
- 法规条文编号

解决方案是混合检索，必要时加入规则引擎。

### 5.8 常见陷阱三：Prompt 太长、目标太多

很多 Prompt 一次性要求模型：

- 理解问题
- 检查合规
- 对比多个版本
- 生成摘要
- 输出表格
- 给出建议
- 生成 JSON

这会导致任务边界混乱，稳定性下降。

更好的方式是：

- 分步处理复杂任务；
- 先检索，再判断，再生成；
- 把审核与回答拆成两个阶段。

### 5.9 常见陷阱四：忽视权限与安全

企业知识库系统常常接入内部文档，因此必须考虑：

- 用户是否有权访问该文档；
- 回答中是否可能泄露敏感信息；
- 日志中是否记录了隐私数据；
- 模型输出是否需要审计。

建议在检索阶段就进行权限过滤，而不是在生成后再“补救”。

### 5.10 常见陷阱五：只做 Demo，不做运维

一个能演示的 RAG 系统，离一个可生产运行的系统还差很多：

- 文档如何增量更新？
- 索引如何重建？
- 向量版本如何兼容？
- Prompt 更新如何灰度发布？
- 错误如何监控？
- 成本如何控制？

上线后真正重要的，往往不是“第一次答得多惊艳”，而是“第 100 万次是否依然稳定、可追溯、可优化”。

## 6. 结论

Prompt Engineering 和知识库构建，并不是两个孤立的话题，而是现代 AI 应用落地中的一体两面。

- **Prompt Engineering** 决定模型是否“听得懂、做得对、答得稳”；
- **知识库构建** 决定模型是否“有资料可依、有依据可查、有边界可控”。

如果你正在建设企业级 AI 问答、智能客服、技术支持助手或内部 Copilot，那么最值得投入的方向通常不是一味追逐更大的模型，而是把下面这条链路打磨扎实：

> 高质量数据 → 合理切分 → 稳定检索 → 清晰 Prompt → 引用回答 → 持续评估

从工程视角看，一个优秀的知识增强型 AI 系统应当具备以下特征：

- 回答有依据，而不是“像是对的”；
- 不确定时会拒答，而不是编造；
- 输出结构稳定，便于集成；
- 可评估、可回归、可持续优化；
- 具备权限、安全和运维能力。

最后可以用一句话总结本文的核心观点：

> **Prompt 决定表达方式，知识库决定事实基础，而最佳实践决定系统能否真正落地。**

如果你刚开始实践，建议从一个小而明确的场景入手：

1. 选择一个垂直问题域；
2. 准备一批高质量知识文档；
3. 设计一个可测试的 Prompt；
4. 建立最小可用的 RAG 闭环；
5. 用真实问题持续评估和迭代。

这样，你构建的就不只是一个“会回答问题的模型”，而是一个真正服务业务、可长期演进的智能知识系统。
